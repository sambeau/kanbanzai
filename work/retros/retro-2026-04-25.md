## Outstanding Issues

### 🔴 H-1 · Shortened plan IDs not accepted by `status()` or `entity()`

**Category:** tool-friction  
**First seen:** P33 session  
**Fixed in:** —  
**Signals:** P33 session retrospective + KE-01KQ275RER55A

Humans naturally refer to plans as `P30`, `P31`, etc. Agents pick up this shorthand from
context. However, `status(id: "P30")` and `entity(action: "get", id: "P30")` both fail — they
require the full canonical ID (e.g. `P30-handoff-skill-assembly-prompt-hygiene`). This causes
agents to error or hunt manually for the full ID every time a human uses the short form.

**Suggestion:** Support shortened plan reference resolution (`P30` → canonical plan ID) in
`status` and `entity` tool routing.

---

### 🔴 H-2 · Tasks start in `queued`, require manual promotion to `ready`

**Category:** tool-friction  
**First seen:** P30, P31 sessions  
**Fixed in:** —  
**Signals:** P31, P30, P32 sessions (3 independent occurrences)

After a feature transitions to `developing`, tasks with no unmet dependencies remain in
`queued` state. They cannot be claimed until manually transitioned to `ready`. This added
multiple round-trips at the start of every development session — for a batch of 13 tasks it was
noticeable friction, reported identically by three separate agents across three separate plans.

**Suggestion:** Tasks with no unmet dependencies should auto-promote to `ready` when the parent
feature enters `developing`.

---

### 🔴 H-3 · `next()` silent timeout leaves task in `active`; cannot be cleanly re-claimed

**Category:** tool-friction  
**First seen:** P30, P31 sessions  
**Fixed in:** Partially — P29 reduced timeout frequency (root cause)  
**Signals:** P31, P30 sessions (2 independent occurrences)

When `next(id: TASK-...)` times out, the task is claimed server-side but the agent receives no
confirmation. On retry the call fails with "already dispatched." The agent has no way to know
the claim succeeded without probing.

**Important:** P29 wired in the SQLite cache and eliminated the O(n) full-directory scans that
caused the *majority* of timeouts. However, the P30 agent encountered this issue **after P29
was already in the binary**, confirming the idempotent-retry UX problem is separate from
the performance fix. The timeout is now rare, but when it does occur, the behaviour is still
confusing.

**Suggestion:** Make task claiming idempotent — a second `next()` on an already-active task
should return the context packet rather than an error, or the timeout error response should
include the claimed task ID so the agent knows the claim succeeded.

### 🔴 H-4 · `decompose` tool proposals produce wrong granularity

**Category:** tool-friction  
**First seen:** P33 session  
**Fixed in:** —  
**Signals:** P33 session (agents bypassed it entirely on two features)

`decompose(propose)` consistently generates one task per acceptance criterion — producing
test-only tasks with no corresponding implementation tasks. A feature with one ~8-line
production change received a nine-task proposal. Agents ended up bypassing the tool entirely
and using `entity(action: create)` directly. The detour through propose → reformat → re-approve
cost more time than it saved.

Note: P8 fixed silent failure modes in `decompose` (spec approval gate, unindexed spec hard
stop) and P24 improved AC pattern recognition, but neither addressed proposal granularity. The
granularity issue is the primary reason agents bypass the tool.

**Suggestion:** Review the decomposition prompt to ensure proposals produce implementation +
test task pairs rather than one task per AC.

### 🔴 H-5 · Working on `main` instead of feature worktrees (recurring)

**Category:** workflow-friction  
**First seen:** P28 era (KE-01KPYJ4TZ88AE)  
**Partially addressed in:** P33 (orchestrate-development skill Phase 0 + Merge Schedule template section)  
**Signals:** KE-01KPYJ4TZ88AE + P31 session (additional occurrence)

This keeps recurring. In P31, worktrees had been created but were not used — all code was
written directly to `main`, leaving the feature branches orphaned and needing manual
reconciliation before `merge(action: execute)` could run.

P33 added a cohort-aware Phase 0 to the `orchestrate-development` skill and a Merge Schedule
section to the dev-plan template. These are skill-level improvements and are in the repo. The
underlying problem — an agent starting work without checking for an existing worktree — is not
prevented by tooling.

**Suggestion:** The `next(id: ...)` context packet should surface the active worktree path when
one exists for the feature, making it visible at claim time rather than requiring a separate
lookup.

### 🟡 M-1 · Duplicate tasks from multiple `decompose` runs clutter the dashboard

**Category:** decomposition-issue  
**First seen:** P31, P32 sessions  
**Fixed in:** —  
**Signals:** P31, P32 sessions

Multiple decompose runs on the same feature leave behind `not-planned` tasks from earlier
rounds mixed in with the active set. In P32/FEAT-3 this produced 12 cancelled tasks alongside
the real 13, making the dashboard show "13/25 done" even when all real work was finished.

P33 added a "decompose quality warning" to the dev-plan template, which hints at the issue, but
does not prevent duplicate task sets from accumulating.

**Suggestion:** `decompose(apply)` should either cancel the previous task set automatically, or
warn when duplicates are detected and offer a bulk-cancel.

### 🟡 M-4 · Complex interface-threading tasks prone to sub-agent context exhaustion

**Category:** tool-friction  
**First seen:** P32, P33 sessions  
**Fixed in:** —  
**Signals:** P32, P33 sessions (2 occurrences)

Tasks that thread a new interface through multiple layers (adding a service dependency, extending
an interface, wiring a gate predicate) are disproportionately prone to sub-agent context
exhaustion. In P32/FEAT-3 this required three dispatch attempts. In P33 a similarly-structured
task stalled twice. Simple contained changes (one-line fixes, renames) complete reliably.

**Suggestion:** Flag interface-threading tasks in decomposition as requiring tighter scoping —
split production code changes from test additions into separate tasks. Add guidance to the
`orchestrate-development` skill.

### 🟡 M-5 · Sub-agent prompt constraints too loose; round-2 review issues trace to implicit assumptions

**Category:** workflow-friction  
**First seen:** P32 rework session  
**Partially addressed in:** P30 (skill routing), P33 (orchestrate-development updates)  
**Signals:** P32-rework session

Multiple round-2 review failures in P32 were caused by sub-agents acting correctly within
loosely specified constraints: `content_hash` was dropped because "hard error or soft-error —
either is fine" didn't flag it as an intentional UX contract; a deliberate-negative test case
was flipped because the agent didn't know its history; a duplicate helper was added because
the agent wasn't told to check for existing equivalents.

P30 and P33 improved skill loading and orchestration guidance, which helps. The prompt-writing
discipline remains a practice issue.

**Suggestion:** Add a prompt checklist to the `orchestrate-development` skill for rework
dispatches: state invariants that must be preserved, identify deliberate-negative test cases,
instruct agents to check for existing equivalents before adding helpers.

## ✅ What Worked Well

### Sub-agent parallelism with disjoint file scopes

Consistently the most effective acceleration technique across every plan. P31 (gate core +
status tool), P30 (pipeline.go + SKILL.md), P32 (5 features across 4 agents), P33 (3 parallel
review tracks) — all returned zero merge conflicts. The key enabler in every case was explicit
file ownership boundaries defined upfront in the dev plan or task delegation prompt. The pattern
is proven and reliable.

*Signals: P31, P32, P32-rework, P33 sessions*

---

### Specification quality enabling implementation

When specs name fields verbatim, prescribe error message wording exactly, and map FR numbers
directly to code decisions, implementation time spent interpreting intent drops to near zero.
P31 was called out explicitly as "the best-case scenario for spec-driven development."

*Signals: P31 session*

---

### Committing state before spawning agents

Pre-dispatch commits prevented any sub-agent git operations from clobbering in-flight state
across all P32 sessions. Zero clobbering across 5 features and 4 parallel agents. A confirmed
practice worth keeping.

*Signals: P32 session*

---

### Re-scoping features at dispatch time

In P32/FEAT-2, reading the spec carefully before writing the agent prompt revealed the
substantive work had already been done in P28/FEAT-1. Reframing the task as a verification
exercise let the agent complete quickly rather than re-implementing completed work.

*Signals: P32 session*

---

### Review process catching AC gutting and test-case flips

In P32-rework, the review caught a sub-agent that had rewritten an acceptance criterion to test
something weaker (classifications exist) instead of the actual spec requirement (content_hash in
response). The test still passed — the failure was only visible on re-reading the AC. A second
review round caught a deliberate-negative test case being silently flipped to positive. Both
were real correctness regressions that would otherwise have shipped.

*Signals: P32-rework session*

---

### Reading existing code before making fixes

Stopping to read `extractToolResultError` and understand the two-path error detection logic
before writing a fix gave the correct solution immediately. Consistent pattern: 5 minutes
reading surrounding code before writing a fix prevents a round-trip.

*Signals: P32-rework session*

---

### `stubDocService` test double pattern

Defining an unexported test double in the test file (`stubDocService` implementing
`merge.DocService`) kept tests self-contained and fast without needing a real document service.
Reusable pattern that matches existing convention in the gate test suite.

*Signals: P31 session*

---

### Integration test design using `CheckGates` directly

Writing integration tests that call `CheckGates` directly with the wired `DefaultGates()` —
rather than requiring a full MCP stack — gave genuine integration coverage without
infrastructure overhead. Confirmed gate wiring, result aggregation, and fail-open paths end-to-end.

*Signals: P31 session*

---

### `advance: true` for multi-stage feature transitions

Reliably walks features through multiple stage gates with prerequisite checking at each step.
Confirmed again across P32. Saves round-trips.

*Signals: KE (original) + P32 confirmation*

---

### Fact-checking sub-agents catching genuine errors

Fact-checking sub-agents found 5 genuine errors in one doc (incorrect conflict risk levels,
missing struct fields) and 3 in another (tool count, bug lifecycle, incident RCA enforcement) —
errors that would have shipped uncorrected. The Check stage in the doc pipeline earns its place.

*Signals: Multiple sessions, confirmed pattern*
