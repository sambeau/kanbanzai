# Specification: Decompose Dev-Plan Registration

## Overview

This specification defines automatic generation, registration, and auto-approval of a
skeleton dev-plan document when `decompose(action: "apply")` successfully creates tasks,
eliminating the need for a manual `doc approve` call to pass the `dev-planning → developing`
gate.

## Problem Statement

This specification covers the automatic registration and auto-approval of a skeleton dev-plan document when `decompose(action: "apply")` successfully creates tasks. It references the parent design document `P28-doc-intel-polish-workflow-reliability/design-p28-doc-intel-polish-workflow-reliability`, §Sprint 2 (Issue 2) and Decision D3.

## Scope

**In scope:**
- Generating and writing a skeleton dev-plan document during `decompose(action: "apply")`.
- Registering the skeleton via `doc(action: "register")` with `type: "dev-plan"` and owned by the feature.
- Auto-approving the document immediately upon creation so the `dev-planning → developing` gate passes without a manual `doc approve` call.
- Idempotency: updating an existing skeleton rather than duplicating it; skipping generation if a non-skeleton dev-plan already exists.
- Including a note in the `decompose(action: "apply")` response when the skeleton is created.

**Out of scope:**
- Changes to `decompose(action: "propose")` or `decompose(action: "review")`.
- Changes to the gate check logic itself.
- Modifying or enriching existing non-skeleton dev-plan documents.
- Any UI or CLI presentation changes beyond the response note.

## Functional Requirements

**REQ-001** — When `decompose(action: "apply")` successfully creates one or more tasks for a feature, the server MUST generate a skeleton dev-plan document, write it to `work/dev-plan/<feature-slug>-decomposed.md`, register it with `type: "dev-plan"` owned by the feature, and immediately mark it `approved`.

**REQ-002** — The skeleton document MUST contain the following sections and fields at minimum:
- A title line: `# Dev Plan: <feature name> (Decomposed)`
- A header table with fields `Feature`, `Created` (ISO date of the apply call), `Method` (`decompose(action: apply)`), and `Status` (`Auto-approved`).
- An `## Overview` section with a brief prose statement that tasks were created via `decompose(action: apply)` and that the document satisfies the `dev-planning → developing` gate.
- A `## Tasks` section containing a Markdown table with columns `Task ID` and `Summary`, one row per created task.

**REQ-003** — If the feature already has a registered dev-plan document that was NOT generated by a prior `decompose apply` call (i.e. a manually authored dev-plan), the server MUST NOT overwrite or replace it. The skeleton generation step is silently skipped and the existing document is left unchanged.

**REQ-004** — If a skeleton dev-plan document already exists for the feature from a prior `decompose apply` call, the server MUST update it (refresh the `## Tasks` table and `Created` date) rather than creating a duplicate registration.

**REQ-005** — The `decompose(action: "apply")` response MUST include a structured note when a skeleton dev-plan is created or updated. The note MUST state the document file path and registered document ID.

**REQ-006** — If `decompose(action: "apply")` fails to create any tasks (zero tasks created), the server MUST NOT generate or register a skeleton dev-plan document.

## Non-Functional Requirements

**REQ-NF-001** — The additional skeleton generation, write, registration, and approval steps MUST complete within 500 ms added latency over the baseline `decompose(action: "apply")` execution time, measured on a warm server under no load.

**REQ-NF-002** — The skeleton document written to disk MUST be valid UTF-8 Markdown and MUST pass the existing document registration validation without errors for `type: "dev-plan"`.

**REQ-NF-003** — Idempotency: calling `decompose(action: "apply")` twice for the same feature with the same task set MUST result in exactly one registered dev-plan document record (no duplicate records).

**REQ-NF-004** — The feature's dev-plan document record MUST be in `approved` status immediately after a successful `decompose apply` call, with no additional `doc approve` invocation required.

## Constraints

- The skeleton MUST NOT be created when `decompose(action: "apply")` results in zero tasks created.
- The skeleton MUST NOT overwrite an existing non-skeleton (manually authored) dev-plan document on the feature.
- No changes may be made to `decompose(action: "propose")` or `decompose(action: "review")` behaviour or response shapes.
- The gate check logic for `dev-planning → developing` MUST NOT be modified; the gate continues to require an approved dev-plan document, and the fix is applied on the `decompose` side.
- Implementation is confined to the `decompose` command handler and its supporting document-management utilities.

## Acceptance Criteria

**AC-001 (REQ-001):** Given a feature in `dev-planning` state with no existing dev-plan document, when `decompose(action: "apply")` is called and successfully creates at least one task, then a file exists at `work/dev-plan/<feature-slug>-decomposed.md`, a document record of `type: "dev-plan"` owned by the feature is registered in the document store, and the record's status is `approved`.

**AC-002 (REQ-002):** Given a skeleton dev-plan document generated by `decompose(action: "apply")`, when the file is read, then it contains a title matching `# Dev Plan: <feature name> (Decomposed)`, a header table with `Feature`, `Created`, `Method`, and `Status` fields, an `## Overview` section, and a `## Tasks` table listing every task created during the apply call with its ID and summary.

**AC-003 (REQ-003):** Given a feature that already has a registered dev-plan document authored manually (not by a prior decompose apply), when `decompose(action: "apply")` is called and creates tasks, then the existing dev-plan document is unchanged, no new dev-plan document record is created for the feature, and no error is returned.

**AC-004 (REQ-004):** Given a feature that already has a skeleton dev-plan document from a prior `decompose apply` call, when `decompose(action: "apply")` is called again (e.g. after adding tasks), then the `## Tasks` table in the existing skeleton is updated to reflect the current task set, the document record count for the feature remains one, and the document status remains `approved`.

**AC-005 (REQ-005):** Given a successful `decompose(action: "apply")` call that creates or updates a skeleton dev-plan, when the response is returned, then the response includes a note field (or equivalent structured field) that contains the file path of the skeleton document and the registered document ID.

**AC-006 (REQ-006):** Given conditions where `decompose(action: "apply")` creates zero tasks (e.g. all proposed tasks already exist), when the call completes, then no dev-plan document file is written and no new document record is registered.

**AC-007 (REQ-NF-003, REQ-NF-004):** Given a feature with no dev-plan, when `decompose(action: "apply")` is called twice sequentially with identical inputs, then exactly one document record of `type: "dev-plan"` exists for the feature, and its status is `approved`.

**AC-008 (REQ-NF-001):** Given a warm server under no load, when `decompose(action: "apply")` is called and triggers skeleton generation, then the total wall-clock time for the call does not exceed the baseline time by more than 500 ms.
