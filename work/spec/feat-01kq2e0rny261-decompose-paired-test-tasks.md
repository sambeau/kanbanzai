## 2. Overview

This specification covers the replacement of the global single catch-all "Write
tests" task in `decompose(action: propose)` output with per-unit paired test
tasks. Each implementation task generated by the decompose proposal algorithm
receives an immediately following sibling test task with an explicit dependency
on its paired implementation, matching scope, and a deterministic slug. The
global catch-all injection is removed. One exception applies: an implementation
task derived solely from an acceptance criterion whose text already contains the
word "test" does not receive a paired test task.

---

## 3. Scope

**In scope:**

- Modification of the task-generation loop in `decompose(action: propose)` to
  append a paired test task immediately after each generated implementation task.
- Removal of the global "Write tests" catch-all task injection.
- The exception rule: ACs containing the word "test" (case-insensitive) do not
  generate a paired test task for their derived implementation task.
- The slug, name, summary, `DependsOn`, and `Covers` fields on the generated
  test tasks.

**Out of scope:**

- Changes to how implementation tasks are grouped (the 1 AC / 2–4 AC grouping
  heuristic is unchanged).
- Changes to `decompose(action: apply)` — it already wires `depends_on` from the
  proposal's `DependsOn` field; no apply-side change is needed.
- Changes to any other field or behaviour of the proposal output not listed above.
- Modification of the section-grouping heuristic or handling of specs with more
  than four ACs per section.

## 4. Functional Requirements

### Paired test task generation

**FR-001** — For each implementation task generated by the decompose proposal
algorithm (whether a single-AC task or a 2–4 AC grouped task), the system MUST
immediately append a sibling test task in the proposal task list.

**FR-002** — The paired test task MUST have the following field values derived
from its implementation task:

- **Slug:** `{impl-slug}-tests` where `{impl-slug}` is the slug of the paired
  implementation task.
- **Name:** `Test {impl-name}` where `{impl-name}` is the name of the paired
  implementation task.
- **Summary:** `Write tests covering: {impl-summary}` where `{impl-summary}` is
  the summary of the paired implementation task.
- **DependsOn:** a list containing exactly one entry: the slug of the paired
  implementation task.
- **Covers:** the same set of AC texts as the paired implementation task.

**FR-003** — The paired test task MUST appear immediately after its paired
implementation task in the proposal task list, before the next implementation
task or the end of the list.

### Exception rule

**FR-004** — When every acceptance criterion covered by an implementation task
contains the word "test" (case-insensitive substring match), the system MUST NOT
generate a paired test task for that implementation task.

**FR-005** — When an implementation task covers multiple ACs and at least one AC
does not contain the word "test", the system MUST generate a paired test task for
that implementation task (the exception applies only when all covered ACs contain
"test").

### Removal of global catch-all

**FR-006** — The system MUST NOT generate a global "Write tests" catch-all task
for the feature. The logic that previously appended a single generic test task
after all implementation tasks MUST be removed.

### Task count

**FR-007** — A feature whose decompose proposal generates N implementation tasks
(none subject to the exception in FR-004) MUST produce exactly 2N tasks in the
proposal output (N implementation tasks and N paired test tasks).

**FR-008** — A feature whose decompose proposal generates N implementation tasks,
of which E are subject to the exception in FR-004, MUST produce exactly
`2N - E` tasks in the proposal output.

## 5. Non-Functional Requirements

**NFR-001** — The `DependsOn` field on paired test tasks MUST use the same slug
format already consumed by `decompose(action: apply)` when wiring `depends_on`
on created tasks. No change to the apply pipeline is required.

**NFR-002** — The case-insensitive "test" substring check (FR-004) MUST be
applied to the AC text as it appears in the spec, with no normalisation beyond
Unicode-aware case folding. The check MUST NOT be applied to task slugs, names,
or summaries — only to the AC text.

## 6. Acceptance Criteria

**AC-001 (FR-001, FR-003, FR-007)** — Given a feature spec with three
implementation-generating ACs (none containing "test"), when
`decompose(action: propose)` is called, then the proposal contains exactly six
tasks, with each test task appearing immediately after its paired implementation
task.

**AC-002 (FR-002)** — Given an implementation task with slug
`implement-entity-cache`, name `Implement entity cache`, and summary
`Add SQLite-backed cache for entity reads`, when the paired test task is
generated, then the test task has slug `implement-entity-cache-tests`, name
`Test Implement entity cache`, summary
`Write tests covering: Add SQLite-backed cache for entity reads`, and
`DependsOn: ["implement-entity-cache"]`.

**AC-003 (FR-002)** — Given the implementation task in AC-002, when the paired
test task is generated, then the test task's `Covers` field contains the same AC
texts as the implementation task's `Covers` field.

**AC-004 (FR-004)** — Given an implementation task derived from a single AC
whose text is `"Add a regression test for the overflow path"`, when
`decompose(action: propose)` is called, then no paired test task is generated for
that implementation task.

**AC-005 (FR-005)** — Given an implementation task derived from two ACs where
one AC text contains "test" and the other does not, when
`decompose(action: propose)` is called, then a paired test task IS generated for
that implementation task.

**AC-006 (FR-006)** — Given any feature spec, when `decompose(action: propose)`
is called, then the proposal output does NOT contain a task with a generic name
such as "Write tests" that covers the entire feature scope without a specific
paired implementation task dependency.

**AC-007 (FR-004)** — Given an implementation task derived from a single AC
whose text is `"Implement the new resolver"` (no "test" substring), when
`decompose(action: propose)` is called, then a paired test task IS generated for
that implementation task.

**AC-008 (FR-008)** — Given a feature spec with four implementation-generating
ACs where one AC contains "test" (exception applies) and three do not, when
`decompose(action: propose)` is called, then the proposal contains exactly five
tasks (four implementation tasks + three paired test tasks).

## 7. Dependencies and Assumptions

**DEP-001** — The `decompose(action: apply)` pipeline already reads the
`DependsOn` field from proposal tasks and wires `depends_on` on created task
entities. No changes to `decomposeApply` are required by this specification.

**DEP-002** — The existing per-AC grouping heuristic (single AC or 2–4 ACs per
task) is the input to the paired-test-task generation step. This specification
assumes that heuristic is stable and does not change it.

**ASM-001** — Implementation task slugs are unique within a proposal. Appending
`-tests` to a unique slug produces a unique test task slug with no collision
risk.

**ASM-002** — The AC text available to the generation algorithm is the raw text
as extracted from the specification document. Case-insensitive substring matching
on this text is sufficient to detect explicit testing ACs.

**ASM-003** — A feature with N implementation tasks produces work that is better
represented as 2N tasks than as N+1 tasks; the increase reflects previously
hidden work, not overhead.
