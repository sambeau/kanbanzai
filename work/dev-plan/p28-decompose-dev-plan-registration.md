# Dev Plan: Decompose Dev-Plan Registration

## Scope

This dev plan covers the implementation of automatic skeleton dev-plan generation, registration,
and auto-approval inside `decompose(action: "apply")`. When `apply` successfully creates one or
more tasks, it must write a skeleton dev-plan file to `work/dev-plan/<feature-slug>-decomposed.md`,
register it as a `type: "dev-plan"` document owned by the feature, and immediately mark it
`approved` — eliminating the need for a manual `doc approve` call to pass the `dev-planning →
developing` gate.

**Specification:** `work/spec/p28-decompose-dev-plan-registration.md`

Scope boundaries:
- In scope: skeleton content generator, `decompose apply` integration, idempotency logic, response note, unit and integration tests.
- Out of scope: changes to `decompose propose` or `decompose review`, gate-check logic, any non-skeleton dev-plan documents.

---

## Task Breakdown

### Task 1: Implement skeleton dev-plan content generator

- **Description:** Implement an unexported helper (e.g. `buildSkeletonDevPlan`) that accepts the feature name, feature ID, apply date, and the list of created tasks (ID + summary pairs), and returns the skeleton Markdown string. The output must conform to REQ-002: a `# Dev Plan: <feature name> (Decomposed)` title, a header table with `Feature`, `Created`, `Method`, and `Status` fields, an `## Overview` prose section, and a `## Tasks` table with `Task ID` and `Summary` columns. The helper must produce valid UTF-8 Markdown that passes document-registration validation for `type: "dev-plan"` (REQ-NF-002).
- **Deliverable:** `buildSkeletonDevPlan` (or equivalent) helper function in `internal/service/decompose.go` (or a new `internal/service/decompose_skeleton.go` file), ready to be called by Task 2.
- **Depends on:** None
- **Effort:** small
- **Spec requirement:** REQ-001, REQ-002, REQ-NF-002

### Task 2: Integrate skeleton generation into `decompose(action: apply)` with idempotency

- **Description:** After tasks are successfully created in `DecomposeApply`, call `buildSkeletonDevPlan` (Task 1) and perform the following steps: (a) Check if the feature already has a registered dev-plan document. If a non-skeleton dev-plan exists (i.e. one not generated by a prior `decompose apply`), skip skeleton generation silently (REQ-003). If a skeleton already exists, overwrite the file and update the existing document record (REQ-004). If no dev-plan exists, write the new file and register + auto-approve it (REQ-001). (b) Write the skeleton Markdown to `work/dev-plan/<feature-slug>-decomposed.md`. (c) Register the document via the document service with `type: "dev-plan"`, `owner: featureID`, and immediately mark it approved (REQ-001, REQ-NF-004). (d) Include a structured note in the `apply` response containing the skeleton file path and document ID (REQ-005). Step (a)–(d) execute only when at least one task was created (REQ-006). The total added latency must not exceed 500 ms on a warm server (REQ-NF-001). Calling `apply` twice with the same inputs must result in exactly one document record (REQ-NF-003).
- **Deliverable:** Updated `DecomposeApply` (or equivalent handler) in `internal/service/decompose.go`, including idempotency logic and the response note field. The `dev-planning → developing` gate must pass without a manual `doc approve` call after this change.
- **Depends on:** Task 1
- **Effort:** medium
- **Spec requirement:** REQ-001, REQ-003, REQ-004, REQ-005, REQ-006, REQ-NF-001, REQ-NF-003, REQ-NF-004

### Task 3: Unit tests for content generator and idempotency logic

- **Description:** Add unit tests in `internal/service/decompose_test.go` (or a new `_skeleton_test.go` file) covering: (1) `buildSkeletonDevPlan` output — assert all required fields and sections are present, the Tasks table lists every provided task, and the Markdown is valid UTF-8 (AC-002, REQ-NF-002). (2) Idempotency — mock document-service responses to simulate: no existing dev-plan (skeleton created), existing skeleton (skeleton updated, record count stays 1), existing non-skeleton dev-plan (skeleton skipped, no new record). (3) Zero-tasks path — assert no file write or registration occurs when task count is zero (AC-006). (4) Response note — assert the apply response contains the skeleton path and doc ID when creation or update occurs (AC-005).
- **Deliverable:** New or updated test file(s) in `internal/service/` with full coverage of the content generator and idempotency branches.
- **Depends on:** Task 1, Task 2
- **Effort:** medium
- **Spec requirement:** REQ-001, REQ-002, REQ-003, REQ-004, REQ-005, REQ-006, REQ-NF-002, REQ-NF-003

### Task 4: Integration test — decompose apply skeleton end-to-end

- **Description:** Add an integration test (or extend an existing one) that exercises the full `decompose apply` path against a real (or near-real) server/store: (1) Create a feature in `dev-planning` state with no dev-plan. Call `decompose apply` and assert: skeleton file exists at `work/dev-plan/<slug>-decomposed.md`, a document record of `type: "dev-plan"` owned by the feature is registered, its status is `approved`, and the response includes the path and doc ID (AC-001, AC-005). (2) Call `decompose apply` a second time; assert exactly one document record exists and status is still `approved` (AC-007). (3) Assert the `dev-planning → developing` gate passes without a manual `doc approve` call (AC-001, REQ-NF-004). (4) Benchmark `decompose apply` with skeleton generation enabled; assert wall-clock delta vs. baseline is ≤ 500 ms (AC-008). (5) Create a feature with a manually authored dev-plan; call `decompose apply`; assert the manual dev-plan is unchanged and no duplicate record exists (AC-003).
- **Deliverable:** Integration test file (e.g. `internal/service/decompose_integration_test.go` or equivalent) that passes in CI without a live git repository of 34+ worktrees.
- **Depends on:** Task 3
- **Effort:** medium
- **Spec requirement:** REQ-001, REQ-003, REQ-004, REQ-NF-001, REQ-NF-003, REQ-NF-004

```
Task 1 (buildSkeletonDevPlan content generator)
    └─► Task 2 (integrate into decompose apply + idempotency)
            └─► Task 3 (unit tests for generator + idempotency)
                    └─► Task 4 (integration test — end-to-end)
```

implementations to be in place to compile. Task 4 requires Task 3 to pass before validating
the full pipeline.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Idempotency logic incorrectly identifies a manually authored dev-plan as a skeleton | Medium | High — could silently overwrite a human document | Distinguish skeleton records via a metadata tag or filename convention (`-decomposed.md` suffix); verify in Task 3 unit tests |
| Document registration latency exceeds 500 ms budget on slow CI runners | Low | Medium — REQ-NF-001 breach | Benchmark in Task 4; gate in CI; use a lightweight in-memory store mock for unit tests |
| Race condition: two concurrent `decompose apply` calls create duplicate records | Low | Medium — violates REQ-NF-003 | Apply a feature-level lock or upsert pattern in the document service; covered by idempotency test in Task 3 |
| Skeleton Markdown fails document-registration validation | Low | High — gate never passes | Validate generated content against the registration schema in Task 3 unit tests before merging |
| Skeleton file path collides with a manually created file of the same name | Low | Low — overwrite risk | Document the `-decomposed.md` naming convention; Task 3 covers the non-skeleton guard |
