# P61 Report — `handoff` Tool Failure: Root Cause Analysis & Remediation

**Status:** investigation complete; Tier 1 stop-gap landed on `main`
**Linked bug:** BUG-01KR45KJWB2KY (`handoff-nil-pipeline-panic`)
**Date:** 2026-05-08
**Author:** architect (investigation session)

---

## 1. Root Cause Finding

**The `handoff` tool was not slow. It was panicking with a nil pointer dereference, and the failure was invisible to the client because the MCP framework's panic recovery suppresses the response.** The client only sees "no reply" and gives up at its read deadline. This was misdiagnosed as a "timeout".

Reproduction (running the actual MCP server against the live `.kbz/`):

```text
panic: runtime error: invalid memory address or nil pointer dereference
[signal SIGSEGV: segmentation violation code=0x2 addr=0x20 pc=0x100cb76ac]

goroutine 42 [running]:
github.com/sambeau/kanbanzai/internal/context.(*Pipeline).stepLookupBinding(0x100e71397?, ...)
	/Users/samphillips/Dev/kanbanzai/internal/context/pipeline.go:303 +0x1c
github.com/sambeau/kanbanzai/internal/context.(*Pipeline).Run(0x0, ...)   ← receiver is nil
	/Users/samphillips/Dev/kanbanzai/internal/context/pipeline.go:226 +0xa4
github.com/sambeau/kanbanzai/internal/mcp.handoffTool.func1(...)
	/Users/samphillips/Dev/kanbanzai/internal/mcp/handoff_tool.go:172 +0x9e0
```

The receiver `Pipeline` is `nil`. The pipeline is `nil` because the binding-file loader rejected `.kbz/stage-bindings.yaml` at server startup. Direct test of the loader against the live file:

```text
DIAG bf nil? true
DIAG errs: 1
DIAG err: decoding binding file: yaml: unmarshal errors:
  line 186: field profile not found in type binding.StageBinding
  line 187: field tier not found in type binding.StageBinding
  line 188: field modes not found in type binding.StageBinding
  line 211: field verifying not found in type binding.StageBinding
```

The `retro-fixing` stage in `.kbz/stage-bindings.yaml` introduced four field shapes (`profile`, `tier`, `modes`, nested `verifying`) that are not present on `binding.StageBinding`. `loader.LoadBindingFile` decodes with `decoder.KnownFields(true)` — strict mode — so it returns `(nil, errs)`.

In `internal/mcp/server.go` the pipeline construction was gated on success, but **the failure path was silent**:

```go
bindingPath := filepath.Join(core.InstanceRootDir, "stage-bindings.yaml")
if bf, errs := binding.LoadBindingFile(bindingPath); bf != nil && len(errs) == 0 {
    ...
    pipeline = &kbzctx.Pipeline{ ... }
    log.Printf("[server] 3.0 context assembly pipeline loaded with %d stage bindings", len(bf.StageBindings))
}
// no else branch; errs dropped on the floor; pipeline stays nil
```

Because `bf == nil`, the pipeline was never constructed. `pipeline` stayed nil. **No log message was emitted** so this silent failure was invisible to the operator.

`handoff_tool.go` then called `pipeline.Run(input)` unconditionally (this used to fall back to a legacy assembly path; commits `21c6040b` and `de75afe6`, "remove legacy assembly path", deleted that fallback in the recent past). It panics at `stepLookupBinding` line 303 (where it dereferences `p`).

The panic was caught by the `mark3labs/mcp-go` worker goroutine but no error response was written back to stdout, so the client (e.g. an editor MCP host) blocked until its own read deadline elapsed. From the caller's vantage point this looked identical to a server-side timeout.

### Hypotheses ruled out (with justification)

| Hypothesis | Verdict | Evidence |
|---|---|---|
| **A. Conflict-detection O(n²) blow-up with 15 active tasks** | **False.** | `handoff_tool.go` and the entire `internal/context/pipeline.go` perform **zero** conflict-detection work. There is no call to `internal/conflict`, no `EntityService.List(active)`, no pair iteration. Conflict checks are an opt-in feature of `next(conflict_check: true)`, not handoff. |
| **B. `doc_intel` index staleness** | **False.** | The handoff pipeline performs **zero** doc_intel calls. There is no spec-section loading, no document classification lookup, no FTS query. The pipeline only reads role YAMLs (`.kbz/roles/*.yaml`, 22 files), skill markdown (`.kbz/skills/*/SKILL.md`, 39 files), the bindings file, and the knowledge store. |
| **C. Code-graph traversal fan-out** | **False.** | Handoff never queries `codebase-memory-mcp`. The "Code Graph" prompt section is a static instructional block referencing the worktree's graph project name — it is rendered text, not a graph query. |
| **D. Unknown — deeper issue** | **Confirmed.** | Reproduced via direct MCP stdio exchange against the running binary and live state. |

The prior diagnosis appears to have been generated by an agent that reasoned from the *description* of what handoff "is supposed to do" rather than from the *actual implementation*. That mismatch is itself a finding worth surfacing (see §6).

**Even with the panic fixed, the warm-state happy-path budget is well under 200 ms** for the current dataset. There is **no scaling problem** in the existing algorithms. The brief's worry about "20, 30, 50 active tasks" does not apply: handoff cost is independent of `len(active_tasks)`.

End-to-end measurement after Tier 1 fix: **587 ms** including server cold start, cache warm-up, and full pipeline run.

### Architectural mistakes that *do* exist (independent of the panic)

These are not the cause of the current outage but should be addressed alongside the fix:

1. **Silent failure of the pipeline constructor in `server.go`.** A core feature is disabled with no log, no health-check signal, and no startup-time error. The `errs` slice from `LoadBindingFile` is discarded. *(Tier 1: addressed.)*
2. **`handoff_tool.go` does not nil-check `pipeline`.** The receiver pointer is reachable from a closure captured at registration time, so the pipeline being `nil` is a static, deterministic crash, not a transient. *(Tier 1: addressed.)*
3. **Strict YAML (`KnownFields(true)`) on a user-edited config file with no schema-migration path.** Adding a new sub-stage block to `.kbz/stage-bindings.yaml` immediately bricks the server. *(Tier 2.)*
4. **`commitStateFunc` uses `exec.Command` with no `context.Context`.** A wedged git process cannot be cancelled by the MCP request deadline. Future timeout vector. *(Tier 2.)*
5. **`Knowledge.Surface` reloads all KEs on every call.** O(KEs) per handoff with no caching. Fine at 140; felt at 5 000. *(Tier 2.)*
6. **`pipeline.Bindings` defensive guards inside steps are useless if the receiver itself is nil.** *(Tier 1: addressed by caller-side nil-guard.)*
7. **MCP handler had no `defer recover()`.** A panic ⇒ no JSON-RPC reply ⇒ client deadlock. *(Tier 1: addressed in handoff; should be applied to all handlers in Tier 2.)*
8. **`entitySvc.Get(parent_feature)` is invoked twice per handoff.** Trivially de-duplicable. *(Tier 2.)*
9. **Doc/comment drift.** `handoff_tool.go` line 7-9 says "Context assembly uses the 3.0 pipeline unconditionally. The pipeline's own validation steps … handle errors." False — they cannot handle errors when the pipeline itself is nil. The brief's description of what handoff "does" (spec sections, conflict annotations, graph traversal) does not match the code. *(Tier 2.)*

## 3. Solution Options

### Tier 1 — Restore service (LANDED)

**S1. Extend `binding.StageBinding` to model the new fields actually in use.** ✅
**S2. Loud startup warnings when binding load fails.** ✅
**S3. Nil-guard `handoff_tool.go` and convert panic to JSON-RPC error.** ✅

Implementation: 65 insertions, 2 deletions across `internal/binding/model.go`, `internal/mcp/server.go`, `internal/mcp/handoff_tool.go`. Verified end-to-end: handoff returns full prompt in 587 ms; no panic; startup log includes `[server] 3.0 context assembly pipeline loaded with 12 stage bindings`.

### Tier 2 — Strengthen the contract (THIS PLAN)

**T1. Schema-versioned stage-bindings file.** Add `schema_version: 2` at top; loader dispatches to the right struct. New fields can be introduced without breaking older binaries; older binaries refuse with a clear error. ~150 LOC + migration script. 1 day.

**T2. Wire `context.Context` through the pipeline and git helpers.** Pass `ctx` from MCP handler into `entitySvc.Get`, `commitStateFunc`, `pipeline.Run`, and ultimately `exec.CommandContext`. Honor a per-tool budget (~5 s). ~200 LOC. 1–2 days.

**T3. Cache loaded knowledge entries with a generation token.** `KnowledgeService` exposes `Generation()` (file-mtime + count, or commit SHA). Surfacer caches the loaded slice and re-reads only when generation changes. ~80 LOC. 4 h.

**T4. Apply the panic-recovery + structured-error pattern to all MCP handlers.** Audit `internal/mcp/*_tool.go` for handlers without `defer recover()` and add the same pattern used in handoff. ~30 LOC per handler × ~15 handlers. 1 day.

**T5. Expose `binding_loadable` as a `health()` check.** Surfaces in dashboards even when serve is up. ~50 LOC. 2 h.

**T6. Documentation reconciliation.** Update `handoff_tool.go` header, `internal/context/pipeline.go` docs, project `AGENTS.md`, and `.github/copilot-instructions.md` to describe what handoff actually does, not the design intent. Decide whether design intent (spec sections, conflict annotations, graph traversal) should be re-introduced as separate features. 4 h + design discussion.

**T7. De-duplicate parent_feature `Get` in handoff handler.** Trivial cleanup. 30 min.

### Tier 3 — Strategic / not warranted now

The brief invites consideration of a different storage engine, async pre-computation, or a different conflict-detection model. **None are warranted by the actual data.** Handoff has no scaling problem. Re-platforming would solve a problem that does not exist. If knowledge entries grow into the tens of thousands, an embedded SQLite + FTS5 store would replace the file-walk loader cleanly — that is a post-2027 concern.

## 4. Recommended Solution

The Tier 1 stop-gap (S1+S2+S3) restored service. The strategic remediation under this plan (T1–T7 above) is detailed in the companion design document `P61-design-handoff-resilience.md`.

Recommended sequencing:
1. **T6 (docs reconciliation)** first — cheap, prevents the next misdiagnosis.
2. **T5 (health check)** + **T4 (panic recovery audit)** next — defence in depth, low risk.
3. **T2 (context propagation)** — touches many call sites, should land before T1.
4. **T1 (schema versioning)** — most invasive; benefits compound over time but risk is highest.
5. **T3 (knowledge cache)** + **T7 (parent_feature dedupe)** — opportunistic.

## 6. Meta-finding (for engineering leadership)

Two systemic concerns surfaced during this investigation are worth recording:

- **A previous orchestration agent produced a detailed, plausible, and entirely fabricated diagnosis** ("conflict-detection O(n²)", "doc_intel staleness", "graph traversal fan-out") for a system whose code does none of those things. The diagnosis was confident enough to anchor the brief that initiated this investigation. This is a model-failure mode worth documenting in retro: when symptoms match generic categories (slow / timeout), agents will pattern-match to generic causes without verifying against source. **Mandate that any complexity-claim diagnosis cite the offending code path** before being accepted as input to a remediation plan.
- **The brief's description of what `handoff` does diverges materially from what the code does.** Spec sections, conflict annotations, and graph traversal are described as integral; none are present in `pipeline.Run`. Either the docs are stale (likely, given the recent "remove legacy assembly path" commits) or the implementation is incomplete vs. the design intent. T6 in the design doc addresses this; an explicit decision is needed about whether the design intent should drive new features or whether the docs should be brought into line with the implementation.
