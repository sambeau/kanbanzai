## Genuine Problems

After filtering out artefacts and one-off issues, the following deserve attention.

### 🔴 P-1 · Commit discipline for workflow/management actions

**Source:** Retro theme #2 (severity 5, KE-01KN5ZXK486TM)  
**Not addressed by:** Any fix batch

The kanbanzai-agents skill's commit checklist is only surfaced when a task is claimed via
`next(id: ...)`. Sessions that do management work (doc registration, entity transitions,
knowledge contributions) without claiming a task never see it. State changes accumulate silently
across a session and are only caught at the start of the next session via the `git status`
pre-task check.

This is not a code bug — it's a structural gap in how the workflow surfaces its own discipline
requirements. B44 (prompt-engineering skill) may improve agent behaviour, but the underlying
issue is that the system has no mechanism to prompt a commit after state-mutating tool calls
outside of a claimed task.

Related knowledge entries describe the same pattern from different angles:
- KE-01KQQ5981YMN0: Files accumulate on main through MCP server commits, WIP-dump commits,
  and direct edits for plans/batches that lack worktree support
- KE-01KQPYAGTW4KT: At session start, `.kbz/` state files are frequently modified or untracked

**Suggestion:** Add an explicit commit step to the kanbanzai-agents skill that covers non-code
changes. Consider having the MCP server auto-commit `.kbz/state/` changes after each
state-mutating tool call when not operating inside a task context.

### 🔴 P-2 · Error taxonomy is completely collapsed

**Source:** Action log instrumentation (7,220 calls, 1,058 failures)  
**Not addressed by:** B35 (Expanded MCP Instrumentation) or any subsequent batch

Every failure across the entire metrics window is classified as `internal_error`. The
`error_type` field in `actionlog.Entry` supports five distinct categories — `gate_failure`,
`validation_error`, `not_found`, `precondition_error`, `internal_error` — but the MCP error
handling never populates them. All failures land in the generic bucket.

This makes the instrumentation investment **non-diagnostic**. You cannot distinguish between:
- "The user made a mistake" (validation_error)
- "The entity doesn't exist" (not_found)
- "A stage gate prerequisite isn't met" (gate_failure)
- "There's a nil pointer dereference" (internal_error)

Without this distinction, the action log can tell you *that* `finish` fails 26% of the time
but not *why*. The `error_type` constants are defined, the `Entry` struct has the field,
the log writer accepts it — it's purely a matter of the MCP handlers not setting it.

**Suggestion:** Audit MCP tool handlers to classify errors at the point they're returned.
Prioritise the high-volume tools: `entity`, `finish`, `next`, `doc`, `decompose`.

### 🔴 P-3 · `decompose` proposal quality and granularity

**Source:** Retro theme #8 (KE-01KN5R87D2TCE) + action log (49.4% failure rate, 81/164 calls)  
**Partially addressed by:** B8, P8, P24 — these fixed AC format parsing and stale-index
fallback, but did not address proposal quality

The decompose tool's persistent problems fall into two buckets:

**Bucket A — now fixed:**
- AC format `**AC-001 (REQ-001):**` was not recognised (fixed in P24, 2026-04-21)
- Stale index produced plausible-but-wrong heading-based tasks (fixed in P24 + P28)

**Bucket B — not fixed:**
- Proposal granularity: one task per AC rather than implementation + test pairs
- Plausible-looking but functionally wrong proposals when fallback triggers
- Circular dependency graphs that assume partial task completion

Agents routinely bypass `decompose propose` and create tasks directly via `entity(action:
create)`. The 49% failure rate persisted through the entire metrics window even after B8/P8/P24
shipped. The AC parsing fixes eliminated the most common *cause* of failures, but the proposal
quality issues mean agents still don't trust the output.

Knowledge entries KE-01KMT4T26W1VZ (stale-index fallback), KE-01KMT4T59DB2N (misleading
fallback output), and KE-01KPVGKJ13QPH (AC format limitation) all document related issues.

**Suggestion:** Review the decompose proposal prompt to produce implementation + test task
pairs. Add a "refuse to propose" mode when the spec cannot be decomposed cleanly, rather than
producing plausible-but-wrong output.

### 🟡 P-4 · The `needs-rework` lifecycle path is undersupported

**Source:** Action log instrumentation  
**Not addressed by:** Any batch

The `needs-rework` stage has the highest failure rate of any normal lifecycle stage: 24.5%
across 49 calls from 19 unique entities. Features that fail review and go back for rework
encounter tooling friction that the happy path doesn't:

- `needs-rework → developing`: The plan-validator already ran before the first `developing`
  entry, but scope changes during rework may invalidate the dev-plan (new tasks, shifted
  dependencies, scope drift). No validator re-runs on re-entry.
- `needs-rework → reviewing`: Skips straight back to review. The review-gate-validator has no
  concept of "second review" — it can't check whether previous findings were addressed.
- The fast-track architecture (P43) models workflow as a pipeline, not a loop. Validators are
  positioned at forward-only stage boundaries. Rework breaks the model.

The stage bindings' `max_review_cycles: 3` and fast-track tiers' `max_auto_cycles` are
separate concepts that don't address the `needs-rework → developing → reviewing` round-trip.

**Suggestion:** At minimum, re-trigger the plan-validator (or checks D3, D4, D5, D6 — the ones
that scope changes can invalidate) on `needs-rework → developing`. Track the rework round-trip
count against the tier's `max_auto_cycles`.

### 🟡 P-5 · Document organization and filename consistency

**Source:** Knowledge entries (KE-01KQSER9N0QHY, KE-01KQQ0RBR7YAP, KE-01KQQ0MPDMYVV)  
**Not addressed by:** Any batch

Document placement conventions exist but are enforced by agent discipline alone — no tooling
validates or generates canonical paths.

**The documented convention** (KE-01KQSER9N0QHY):
- Documents live in `work/{plan-slug}/{plan-id}-{type}-{topic}.md`
- Each plan folder is self-contained
- `work/research/` is deprecated

**The gaps:**
- Filename variations creep in (e.g. `P41-prompt-design-review.md` vs. standard naming)
- Prompt files have no defined home — scattered based on individual judgment
- No deterministic tool generates the correct path given a document type and entity context
- The `doc(action: register)` tool accepts any path; there's no path validation

This is not causing workflow failures, but it adds cognitive overhead to every session —
agents must remember or rediscover the convention, and inconsistent naming makes it harder to
find documents later.

**Suggestion:** Add a `path` action to the `doc` tool that takes a type (design, spec,
research, prompt) and optional parent entity ID and returns the canonical path. Add a
path-conformance check to document registration. Define where prompt files belong.

### 🟡 P-6 · Worktree adoption and main-branch drift

**Source:** Retro theme #7 (severity 3, KE-01KPYJ4TZ88AE) + KE-01KQQ5981YMN0  
**Partially addressed by:** B39 (worktree dev experience), P42 (edit_file entity_id)

B39 made worktrees work better and P42 made `edit_file` work inside them, but the behavioural
pattern of agents drifting to `main` isn't fully solved. The `next()` context packet doesn't
surface the active worktree path — it requires a separate `worktree(action: get)` call.
And plans/batches have no worktree support at all (KE-01KQQ5981YMN0), so plan-level work
always happens on main.

**Suggestion:** Surface the active worktree path in the `next()` context packet when one exists
for the feature. Consider worktree isolation for plan/batch entities.

### 🟢 P-7 · `entity.transition` remains the dominant operation

**Source:** Action log — 933 calls, the single most common tool+action pair  
**Partially addressed by:** B18 (workflow automation), B34 (agent workflow ergonomics)

Despite automation improvements, manual state transitions are still the #1 activity — more than
reading files (473 `write_file` calls) or checking status (429 `status` calls). B34 added
auto-promotion features, but the metrics suggest they haven't fully landed in practice.

This may self-resolve as fast-track (P43) matures — automated gate validation should reduce
the number of manual transitions needed.
