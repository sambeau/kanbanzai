### Design-led workflow

The Kanbanzai System is design-led: it manages the process of getting from a proposal through a design process that outputs a *specification*.

This is achieved through a series of design document revisions (drafts) until a final design is agreed and approved.

An approved design moves to the specification stage and a specification document is created.

The design manager can request changes to a specification before approving it.

Once a specification is approved, the design manager has completed their role in the process. They then effectively become a product manager and manage the initiation of the development process. I.e. the only role a human has in implementation of a feature is to choose when the implementation should begin.

The design process is an agile development process because: as long as the implementation hasn’t started, a design manager can request a feature be returned to any previous stage of the workflow.

The design process is an agile design  process: as it embraces iteration. An epic (or even feature) can go through multiple development cycles: additions (amendments) are made to the documents, a new round of specifications, planning and implementation can take place.

### Document-led process

The main human interface for the Kanbanzai process is text written in Markdown documents.

In particular:
- Proposals
- Research
- Design
- Specification

#### Proposal documents
The process starts with a proposal. This can be a document or a proposal written in a chat message. The proposal kicks of a discussion that may go nowhere, but normally ends with a design draft.

The system will store, but does not *manage* Proposal documents as they are not a formal stage in the workflow: workflow begins with the first design document.

#### Research documents

The process of creating a design draft may include 
- research: online research that ends in a research document
- reports: local research of the codebase, current documents or processes

The system will store, but does not *manage* Research and Report documents as they are not a formal stage in the workflow: workflow begins with the first design document.

#### Design Documents

A design document can go through as many drafts as is needed before becoming a final design.

Throughout the design process the human design manager and the AI Agent acting as senior designer discuss and hone the design by making decisions. Gradually alternative designs are dropped until one tree design remains: this is the final design.

The final design should be unambiguous and  should not contain alternative designs unless needed to inform the user documentation. The final design is used to create a specification document *and* to create user documentation from.

Once approved, the Senior Designer (AI Agent) uses the final design to create a specification document.

The system manages design document as they are part of the workflow system.

#### Specification documents 

A specification document is a *design specification* (not an implementation specification): it is a formalised, unambiguous distillation of the design. It is used to inform the creation of an implementation, tests and is the blueprint used when reviewing an implementation: the main purpose of a review is to check the implementation *against* the blueprint.

Specification documents are created by the Senior Designer (an AI Agent role).

Specification documents are checked and approved by the Design Manager (human).

Once a specification is approved, the design process is finished: the design manager’s job is finished and the work is handed to the development team. The dev team then create an implementation plan from it.

### Specification-led implementation

Kanbanzai is a specification-led implementation process: in effect, a waterfall development process. 

The implementation uses the specification to create an implementation plan that orchestrates the implementation by a team of AI Agents.

Once a pan is written, the dev team waits for the a human to request implementation.

### Chat-based AI Agent project manager

In the Kanbanzai System the human is the Design (and Product) Manager: they design what will be built and when.

The AI agent fills in the other roles: project manager, senior designer, development manager and, of course, all the roles in the development team.

Communication between Design and Product Manager (the human) and the rest of the team (AI Agents) is done through chat. In most cases the human in design manager role is discussing design with the senior designer role or the project in general with the AI Agent acting as project manager. 

The discussion takes places through chat messages.

Chat is agile and flexible and allows for more natural workflow than a command based system or web app.
