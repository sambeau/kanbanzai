# ID System Specification

- Status: specification draft
- Purpose: define the required behavior, constraints, and acceptance criteria for the Kanbanzai identity system
- Date: 2026-07-24
- Based on:
  - `work/design/id-system-design.md`
  - `work/spec/phase-1-specification.md` §13 (identity requirements)

---

## 1. Purpose

This specification defines the identity system for all Kanbanzai workflow entities. It replaces the identity requirements in the Phase 1 specification (§13) with a complete, testable definition.

The ID system must solve one problem: every entity needs a unique, stable, permanent identifier that works without coordination between agents, worktrees, or branches.

This specification defines **what the ID system must do**. The design rationale is in `work/design/id-system-design.md`.

---

## 2. Scope

This specification covers:

- ID formats for all entity types
- the TSID13 encoding
- canonical form and storage rules
- display conventions and prefix matching
- filename format
- epic slug rules
- collision handling
- migration from sequential IDs

This specification does not cover:

- concurrency controls beyond ID uniqueness (separate concern)
- agent identity or user identity
- external system integration IDs

---

## 3. Entity ID Formats

### 3.1 Epics

Epic IDs use human-chosen slugs.

- Format: `EPIC-{SLUG}`
- The slug is chosen by a human at creation time.
- The slug must conform to the rules in §8.
- Uniqueness is enforced at creation time by checking existing epics.

### 3.2 Features

- Format: `FEAT-{TSID13}`
- The TSID13 portion is system-allocated per §5.

### 3.3 Bugs

- Format: `BUG-{TSID13}`
- The TSID13 portion is system-allocated per §5.

### 3.4 Decisions

- Format: `DEC-{TSID13}`
- The TSID13 portion is system-allocated per §5.

### 3.5 Tasks

- Format: `TASK-{TSID13}`
- The TSID13 portion is system-allocated per §5.
- Tasks have independent IDs. The parent feature relationship is stored in entity metadata (`parent_feature` field), not embedded in the ID.

### 3.6 Documents

- Format: `DOC-{TSID13}`
- The TSID13 portion is system-allocated per §5.

### 3.7 Format summary

| Entity   | Format          | Example               | ID length |
|----------|-----------------|-----------------------|-----------|
| Epic     | `EPIC-{SLUG}`   | `EPIC-IDS`            | Variable  |
| Feature  | `FEAT-{TSID13}` | `FEAT-01J3K7MXP3RT5`  | 18 chars  |
| Bug      | `BUG-{TSID13}`  | `BUG-01J4AR7WHN4F2`   | 17 chars  |
| Decision | `DEC-{TSID13}`  | `DEC-01J3KABCDE7MX`   | 17 chars  |
| Task     | `TASK-{TSID13}` | `TASK-01J3KZZZBB4KF`  | 18 chars  |
| Document | `DOC-{TSID13}`  | `DOC-01J3K7MXP3RT5`   | 17 chars  |

## 4. Required Properties

All entity IDs must be:

1. **Unique** — no two entities of any type share the same ID.
2. **Stable** — an ID never changes after creation.
3. **System-allocated** — IDs for TSID-based entities are generated by the system. Users must not manually construct or edit them.
4. **Safe for concurrent work** — two agents creating entities independently, on the same machine or different worktrees, must not produce colliding IDs.
5. **Time-sortable** — TSID-based IDs sort lexicographically by creation time.
6. **URL-safe** — IDs contain only characters safe for use in URLs, filenames, and YAML values without quoting.

## 5. TSID13 Encoding

### 5.1 Structure

A TSID13 is a 13-character string with two components:

| Component | Characters | Bits | Content |
|-----------|------------|------|---------|
| Timestamp | 1–10       | 48   | Milliseconds since Unix epoch (unsigned) |
| Random    | 11–13      | 15   | Cryptographically secure random value |
| **Total** | **13**     | **63** | 65 encoded bits, 2 unused (set to zero) |

### 5.2 Encoding alphabet

TSID13 uses the Crockford base32 alphabet:

```
0123456789ABCDEFGHJKMNPQRSTVWXYZ
```

This alphabet excludes I, L, O, and U to avoid visual ambiguity.

### 5.3 Canonical case

The canonical encoding is uppercase. The system must:

- produce uppercase output in all stored forms
- accept case-insensitive input
- normalise input to uppercase before storage or lookup

### 5.4 Timestamp

The timestamp is the number of milliseconds since the Unix epoch (1970-01-01T00:00:00Z).

It is encoded as the most significant bits of the first 10 Crockford base32 characters. The top 2 bits of the first character are zero for all dates before approximately year 10889.

### 5.5 Random component

The random component is 15 bits generated from a cryptographically secure source (`crypto/rand` or equivalent).

Each ID generation must produce a fresh random value. There is no monotonic increment within the same millisecond.

### 5.6 Sortability

TSID13 values must sort lexicographically in chronological order. IDs created later must sort after IDs created earlier.

Two IDs created within the same millisecond have undefined relative order (the random component determines their sort position).

## 6. Collision Handling

### 6.1 Collision model

- Two IDs generated more than 1 millisecond apart are guaranteed unique (different timestamps).
- Two IDs generated within the same millisecond have a 1-in-32,768 collision probability per pair (15 bits of randomness).

### 6.2 Local collision check

On ID generation, the system must check whether the generated ID already exists in the local store.

If a collision is detected:

1. Regenerate the random component.
2. Retry the collision check.
3. If the collision persists after 3 retries, return an error.

This check is a safety net. It eliminates collisions within a single process.

### 6.3 Cross-worktree collisions

When agents work in separate Git worktrees, there is no shared store to check against. The statistical guarantees of §6.1 apply — collision probability is negligible for any realistic workflow.

If a cross-worktree collision is detected at merge time (duplicate filenames), it must be resolved manually. The system is not required to detect or auto-resolve this case.

## 7. Canonical Form

### 7.1 Definition

The canonical form of an ID is the format used in all persistent storage: YAML entity files, filenames, cross-references in entity fields, and commit messages.

The canonical form does **not** include the break hyphen (§9.2).

Examples of canonical form:

- `EPIC-IDS`
- `FEAT-01J3K7MXP3RT5`
- `BUG-01J4AR7WHN4F2`
- `TASK-01J3KZZZBB4KF`

### 7.2 Where canonical form is required

The canonical form must be used in:

- the `id` field of entity YAML files
- all cross-reference fields (`parent_feature`, `epic`, `affects`, `origin_feature`, `origin_task`, `supersedes`, `superseded_by`, `fixed_by`, etc.)
- entity filenames
- commit messages
- any other persistent or machine-readable context

### 7.3 Case normalisation

The canonical form is uppercase. If the system receives a lowercase or mixed-case ID as input, it must normalise to uppercase before storage or comparison.

## 8. Epic Slug Rules

Epic slugs must satisfy all of the following:

1. Length: 2–20 characters.
2. Character set: uppercase ASCII letters (`A-Z`), digits (`0-9`), and hyphens (`-`).
3. No leading hyphen.
4. No trailing hyphen.
5. No consecutive hyphens.
6. URL-safe (this is satisfied by rule 2).
7. Unique within the project (enforced at creation time).

Input normalisation:

- Lowercase input is accepted and converted to uppercase.
- Spaces are rejected, not silently converted to hyphens.

Invalid slugs must be rejected with an error that explains which rule was violated.

## 9. Display Conventions

### 9.1 Three forms

Every TSID-based ID has three display representations:

| Form | Purpose | Example |
|------|---------|---------|
| Canonical | Storage, filenames, references | `FEAT-01J3K7MXP3RT5` |
| Full display | When the complete ID must be shown | `FEAT-01J3K-7MXP3RT5` |
| Short display | Listings, conversation, human interaction | `FEAT-01J3K-7MX` |

Epic IDs have only one form (the slug is already human-readable).

### 9.2 The break hyphen

The break hyphen is a display-only convention. It is inserted after the fifth TSID character in the full display and short display forms.

The break hyphen must not be stored in YAML files, filenames, entity references, or any persistent format.

The system must strip break hyphens from input before processing. Users may include or omit them freely.

### 9.3 Short display length

The short display form shows the shortest prefix of the TSID that is unique among all entities of the same type within the project.

The short display length is computed at display time, not stored. It changes as entities are added to the project.

The minimum short display length is 5 characters (the portion before the break hyphen position).

## 10. Prefix Matching

### 10.1 Required behavior

The CLI and MCP operations must accept any unique prefix of an entity ID in place of the full canonical ID.

### 10.2 Resolution rules

Given a user-provided ID prefix:

1. Strip the type prefix (e.g., `FEAT-`) and all hyphens from the input to yield the TSID prefix.
2. Normalise to uppercase.
3. Find all entities of that type whose canonical TSID starts with the given prefix.
4. If exactly one entity matches: resolve to that entity.
5. If multiple entities match: return all candidates and indicate that the prefix is ambiguous.
6. If no entity matches: return an error.

### 10.3 Type prefix is required

The type prefix (`FEAT-`, `BUG-`, `TASK-`, `DEC-`, `DOC-`) is always required in prefix matching. A bare TSID fragment without a type prefix must be rejected.

Epic IDs are matched by slug prefix using the same uniqueness rules.

## 11. Filename Format

### 11.1 Entity files

Entity files must use the format:

```
{CANONICAL-ID}-{slug}.yaml
```

Examples:

- `EPIC-IDS-id-system-redesign.yaml`
- `FEAT-01J3K7MXP3RT5-profile-editing.yaml`
- `TASK-01J3KZZZBB4KF-implement-id-allocator.yaml`

### 11.2 Parsing rules

The filename parser must:

1. Strip the `.yaml` extension.
2. Extract the type prefix (everything before the first hyphen).
3. For TSID-based types: the next 13 characters after the prefix hyphen are the TSID; the remainder after the subsequent hyphen is the slug.
4. For epics: everything between `EPIC-` and the slug boundary is the epic slug. The epic slug is the ID itself, so the slug portion of the filename is the human-readable name, separated from the ID by a hyphen after the epic slug.

### 11.3 Slug in filenames

The slug portion of the filename is a human-readable label for filesystem navigation. It is derived from entity metadata (the `slug` field) and must not be treated as part of the ID.

---

## 12. Provenance

Creator identity is not part of the ID.

Provenance is tracked through:

1. Entity metadata fields (`created_by`, `reported_by`, `decided_by`).
2. Git commit history.

The system must not embed provenance information in IDs.

## 13. Migration

### 13.1 From sequential IDs

The sequential ID allocator (`FEAT-001`, `BUG-002`, etc.) is replaced by the TSID13 allocator.

### 13.2 Code changes required

The following components must be updated:

1. **ID allocator** (`internal/id`): replace scan-max-increment with TSID13 generation for Features, Bugs, Decisions, Tasks, and Documents. Replace sequential allocation with slug-based allocation for Epics.
2. **Storage filename parser** (`internal/storage`): accept the new TSID13 filename format. Retain recognition of legacy sequential IDs for robustness during transition.
3. **Service layer** (`internal/service`): use the new allocator for entity creation. Add prefix resolution for get and update operations.
4. **Document ID allocation** (`internal/document`): replace the in-memory counter with TSID13 generation using the `DOC` prefix.
5. **Validation** (`internal/validate`): update ID format validation to accept TSID13-based IDs. Validate epic slugs against §8 rules.
6. **Display layer**: implement break hyphen insertion and shortest-unique-prefix computation.

### 13.3 Unified allocator interface

The system must provide a single allocator interface that accepts an entity type and returns a correctly formatted ID.

- For entity type **Epic**: the allocator accepts a human-provided slug and returns an `EPIC-{SLUG}` ID after validation per §8.
- For all other entity types (**Feature**, **Bug**, **Decision**, **Task**, **Document**): the allocator generates a TSID13 and returns a `{TYPE}-{TSID13}` ID.
- The allocator must enforce the collision check (§6.2) for TSID-based types and the uniqueness check for epic slugs.
- Callers must not construct IDs manually or implement type-specific allocation logic.

### 13.4 Task field rename

The Task entity's `feature` field is renamed to `parent_feature` to reflect that task IDs are now independent (not embedded in the feature ID).

This rename affects:

1. **Entity model** (`internal/model`): the struct field tag changes from `yaml:"feature"` to `yaml:"parent_feature"`.
2. **Validation** (`internal/validate`): any rules referencing the `feature` field must use the new name.
3. **Service layer** (`internal/service`): task creation and update code must use `parent_feature`.
4. **MCP and CLI interfaces**: input parameters and output fields must use `parent_feature`.
5. **Decision log** (P1-DEC-009): the Task minimum fields table must be updated to list `parent_feature` instead of `feature`.
6. **Phase 1 specification** (§9.3): the Task minimum fields list must be updated.

No data migration is required because §13.7 confirms no production state exists.

### 13.5 Test fixture migration

All test fixtures and test data must be updated to use TSID13-format IDs.

Specifically:

1. Test fixtures in `testdata/` directories that use sequential IDs (`FEAT-001`, `BUG-002`, `TASK-001.1`, etc.) must be rewritten with TSID13-format IDs.
2. Test helpers that construct entity IDs must use the allocator or well-known test constants in TSID13 format.
3. Tests must not hard-code sequential-format IDs in assertions unless explicitly testing legacy ID recognition (§13.6).
4. At least one test must verify that the system correctly reads a file with a legacy sequential ID without error (covering the §13.6 robustness guarantee).

### 13.6 Legacy ID recognition

The system should continue to recognise legacy sequential IDs (`FEAT-001`, `BUG-002`) in stored files for robustness. It must not generate new sequential IDs.

### 13.7 Timing

Migration happens before production state exists. There is no existing entity data to convert.

## 14. Acceptance Criteria

The ID system implementation is acceptable only if all of the following are true.

### 14.1 ID generation

It must be possible to create entities of all six types (Epic, Feature, Bug, Decision, Task, Document) and receive a valid ID conforming to the format in §3.

### 14.2 TSID13 correctness

Generated TSID13 values must:

- be exactly 13 characters long
- use only characters from the Crockford base32 alphabet
- be uppercase
- sort lexicographically in chronological order when generated more than 1 millisecond apart

### 14.3 Uniqueness

Generating 10,000 IDs of the same type in a tight loop must produce no duplicates.

### 14.4 Collision check

If the system detects a generated ID already exists in the local store, it must regenerate and retry without returning an error to the caller (up to the retry limit in §6.2).

### 14.5 Epic slug validation

The system must reject epic slugs that violate any rule in §8:

- too short (`"A"`) or too long (21+ characters)
- containing invalid characters (`"MY EPIC"`, `"my_epic"`)
- leading, trailing, or consecutive hyphens (`"-EPIC"`, `"EPIC-"`, `"EPIC--NAME"`)

The system must accept valid slugs and normalise lowercase input to uppercase.

### 14.6 Canonical storage

Entity files, YAML cross-references, and filenames must use the canonical form (§7) — no break hyphens in stored data.

### 14.7 Prefix matching

Given a set of entities, the system must:

- resolve an unambiguous prefix to the correct entity
- return all candidates for an ambiguous prefix
- return an error for a prefix that matches nothing
- accept and strip break hyphens in input
- perform case-insensitive matching

### 14.8 Display conventions

The CLI and any display layer must:

- insert the break hyphen at TSID character position 5 in full and short display forms
- compute the shortest unique prefix per entity type for short display
- never include the break hyphen in stored or persistent output

### 14.9 Filename format

Entity files must use the `{CANONICAL-ID}-{slug}.yaml` format. The filename parser must correctly extract the ID and slug from filenames of all entity types.

### 14.10 Round-trip integrity

Writing an entity to disk, reading it back, and writing it again must produce identical files. The ID must survive the round-trip unchanged.

### 14.11 Time sortability

Creating two entities of the same type with a delay of at least 2 milliseconds between them must produce IDs where the second sorts lexicographically after the first.

### 14.12 Unified allocator routing

- Calling the allocator with each of the six entity types must return an ID in the correct format for that type.
- Calling the allocator with an invalid entity type must return an error.
- No code path outside the allocator may construct entity IDs.

### 14.13 Test fixture consistency

All test fixtures must use TSID13-format IDs. No test outside of explicit legacy-recognition tests may depend on sequential ID formats. At least one test must verify that a legacy sequential ID (`FEAT-001`) is recognised without error.

---

## 15. Out-of-Scope Clarifications

The following are explicitly not part of this specification:

- Optimistic locking or concurrency controls beyond ID uniqueness — these are a separate concern addressed by the concurrency design.
- Cross-worktree collision auto-resolution — collision at merge time is resolved manually.
- External system ID mapping (e.g., GitHub issue numbers) — deferred to later phases.
- ID rotation, revocation, or reuse — IDs are permanent. There is no mechanism to retire or reassign an ID.
